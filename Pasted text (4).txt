Pipeline V3: Hermes Agent + Qwen 3.5 + GLM-OCR
Context
Replace the current hardcoded Python orchestrator (intent.py regex → orchestrator.py dispatch → legacy_services) with an agentic tool-calling loop where Qwen 3.5 27B IS the brain that decides what tools to call. Claude CLI subprocess is eliminated. DeepSeek-OCR (3B) replaced with GLM-OCR (0.9B). Frontend unchanged — same NDJSON streaming contract.

The key shift: Python code decides → LLM executes → Python continues becomes LLM decides → calls tools (Python + OCR) → LLM continues. Pipeline is emergent from system prompt + tools, not hardcoded handlers.

Precondition Enforcement (Gates with Absolute Authority)
Critical design rule: Agent = planner with unreliable judgment. Python tools = gates with absolute authority. If agent can bypass gates → system degrades. If agent is forced through gates → system stabilizes.

Specific enforcement rules:

Tools enforce preconditions, not the LLM — auto_map_tokens fails hard if state != HTML_READY+
Mapping completeness: unresolved_tokens == 0 AND semantic checks pass — confidence is NOT a gate. See Mapping Sanity section below.
Contract sanity is structural — validate_pipeline checks joins produce rows, reshape produces expected cardinality, computed fields not degenerate
Validation is blocking — generate_report checks "validate" in session.completed_stages. If validate_pipeline returned fail, generation is impossible. Agent CANNOT say "good enough."
State transitions enforced by session.py — session_transition("GENERATING") raises ValueError if current state doesn't allow it
Retry limits enforced in Python — ctx.retry_counts["validate"] <= 2, not system prompt. Hard stop in tool layer.
Multi-Candidate Comparative Mapping (Pre-Contract Gate)
Root cause of plausible wrongness: Single-shot mapping picks one answer. If that answer maps row_set_wt_kg → bin1_act instead of bin1_sp, it passes all structural checks because both are INTEGER columns with similar distributions. Subtle swaps survive.

Solution: Generate 2-3 candidate mappings, score each on 3 axes, pick best OR ask user if scores are close. This runs inside tool_auto_map_tokens and tool_resolve_mapping_pipeline, BEFORE build_contract is ever called.

How It Works
Step 1: LLM generates 3 candidate mappings (temperature=0.7, n=3)
Step 2: Python scores each candidate on 3 axes (no LLM needed)
Step 3: Pick winner OR flag ambiguity for user
Step 4: Semantic sanity checks on winner
Step 5: Only then allow build_contract
Step 1: Multi-Candidate Generation
Current run_mapping_preview() calls Qwen once with temperature=0 → single deterministic mapping.

Change: Call Qwen 3 times with temperature=0.7 (or use n=3 if vLLM supports it). Each call gets the same prompt (tokens + schema + OCR context) but produces different mappings due to sampling.

async def _generate_candidate_mappings(
    tokens: list[str], 
    schema: dict, 
    ocr_context: str | None,
    n_candidates: int = 3,
) -> list[dict]:
    """Generate n candidate mappings via LLM with temperature sampling."""
    candidates = []
    for i in range(n_candidates):
        mapping = await _call_mapping_llm(
            tokens=tokens,
            schema=schema,
            ocr_context=ocr_context,
            temperature=0.0 if i == 0 else 0.7,  # first is deterministic, rest sampled
        )
        candidates.append(mapping)
    
    # Deduplicate identical mappings
    unique = _deduplicate_mappings(candidates)
    return unique  # 1-3 unique candidates
Step 2: Scoring (Python, No LLM)
Each candidate scored on 3 axes. Each axis produces a 0.0-1.0 score per token. Final score = weighted average across all tokens.

Axis 1: Token-Column Name Similarity (0.0-1.0)

Fuzzy string matching between token name and column name. Catches obvious swaps.

def _score_name_similarity(token: str, column: str) -> float:
    """
    row_set_wt_kg ↔ bin1_sp → low (0.2)
    row_set_wt_kg ↔ set_weight → high (0.8)
    row_material_name ↔ bin1_content → medium (0.4, "content" ~ "material")
    """
    from difflib import SequenceMatcher
    
    # Normalize: strip row_ prefix, expand abbreviations
    tok_norm = _normalize_token(token)   # "row_set_wt_kg" → "set weight kg"
    col_norm = _normalize_column(column) # "bin1_sp" → "bin 1 setpoint"
    
    # Direct ratio
    ratio = SequenceMatcher(None, tok_norm, col_norm).ratio()
    
    # Bonus: semantic alias matches (sp↔setpoint↔set_weight, act↔actual↔achieved)
    ALIASES = {
        "sp": ["set", "setpoint", "set_weight", "target"],
        "act": ["actual", "achieved", "ach", "measured"],
        "content": ["name", "material", "description", "label"],
    }
    for key, aliases in ALIASES.items():
        if key in col_norm and any(a in tok_norm for a in aliases):
            ratio = max(ratio, 0.7)
    
    return ratio
Axis 2: Column Type Alignment (0.0 or 1.0, hard)

Binary check: does the token's expected type match the column's actual type?

def _score_type_alignment(token: str, column: str, schema: dict) -> float:
    """
    row_set_wt_kg → INTEGER/REAL column → 1.0
    row_set_wt_kg → TEXT column → 0.0
    row_material_name → TEXT column → 1.0
    row_material_name → INTEGER column → 0.0
    """
    col_type = schema.get_column_type(column)
    expected = _infer_token_type(token)  # "numeric", "text", "datetime"
    
    TYPE_COMPAT = {
        "numeric": {"INTEGER", "REAL", "FLOAT", "DOUBLE", "NUMERIC", "DECIMAL"},
        "text": {"TEXT", "VARCHAR", "CHAR", "BLOB"},
        "datetime": {"TEXT", "DATETIME", "TIMESTAMP", "DATE"},  # TEXT ok for dates in SQLite
    }
    
    return 1.0 if col_type in TYPE_COMPAT.get(expected, set()) else 0.0
Axis 3: Distribution Alignment (0.0-1.0)

Sample 10 rows from mapped column. Compare distribution characteristics against token expectations.

def _score_distribution_alignment(token: str, column: str, loader, table: str) -> float:
    """
    row_set_wt_kg → column with values [50, 100, 80, 30, 150] → 1.0 (numeric, reasonable weights)
    row_set_wt_kg → column with values [50, 100, 80, 30, 150] but this is bin_act not bin_sp → ???
    
    Can't distinguish bin_sp from bin_act by distribution alone (similar ranges).
    BUT can distinguish:
    - row_material_name → [50, 100, 80] (numeric where text expected) → 0.0
    - row_sl_no → [1, 2, 3, 4, 5] (sequential integers) → 1.0
    - row_start_time → ["2025-10-01 00:01:45"] (datetime string) → 1.0
    - row_batch_no → [2459, 2460, 2461] (monotonic integers) → 0.8
    """
    sample = loader.frame(table).head(10)
    col = column.split(".", 1)[1] if "." in column else column
    if col not in sample.columns:
        return 0.0
    
    values = sample[col].dropna()
    if values.empty:
        return 0.0
    
    score = 0.5  # neutral baseline
    
    expected = _infer_token_type(token)
    
    if expected == "numeric":
        numeric_count = sum(1 for v in values if _is_numeric(v))
        score = numeric_count / len(values)
    elif expected == "text":
        text_count = sum(1 for v in values if isinstance(v, str) and not _is_numeric(v))
        score = text_count / len(values) if len(values) > 0 else 0.0
    elif expected == "datetime":
        dt_count = sum(1 for v in values if _looks_like_datetime(str(v)))
        score = dt_count / len(values)
    
    # Bonus: sequential check for sl_no/index tokens
    if "sl_no" in token.lower() or "index" in token.lower():
        if _is_sequential(values):
            score = min(score + 0.3, 1.0)
    
    return score
Step 3: Candidate Selection
def _select_best_mapping(
    candidates: list[dict],  # [{token: column, ...}, ...]
    tokens: list[str],
    schema: dict,
    loader,
    table: str,
) -> tuple[dict, bool]:
    """
    Returns (best_mapping, needs_user_confirmation).
    needs_user_confirmation=True when top 2 candidates are within 0.1 of each other.
    """
    WEIGHTS = {"name": 0.4, "type": 0.3, "distribution": 0.3}
    
    scored = []
    for candidate in candidates:
        total = 0.0
        token_scores = {}
        for token in tokens:
            col = candidate.get(token, "UNRESOLVED")
            if col == "UNRESOLVED":
                token_scores[token] = 0.0
                continue
            
            s_name = _score_name_similarity(token, col)
            s_type = _score_type_alignment(token, col, schema)
            s_dist = _score_distribution_alignment(token, col, loader, table)
            
            weighted = (s_name * WEIGHTS["name"] + 
                       s_type * WEIGHTS["type"] + 
                       s_dist * WEIGHTS["distribution"])
            token_scores[token] = weighted
            total += weighted
        
        avg = total / len(tokens) if tokens else 0.0
        scored.append({"mapping": candidate, "score": avg, "token_scores": token_scores})
    
    scored.sort(key=lambda x: x["score"], reverse=True)
    
    best = scored[0]
    runner_up = scored[1] if len(scored) > 1 else None
    
    # Close race → ask user
    needs_confirmation = (
        runner_up is not None 
        and abs(best["score"] - runner_up["score"]) < 0.1
    )
    
    return best, runner_up, needs_confirmation
Step 4: Semantic Sanity Checks (on winner)
After selecting the best candidate, run hard sanity checks. These are binary pass/fail gates.

Cross-field consistency — tokens that MUST map to different columns:

set_wt and ach_wt → same column? REJECT
start_time and end_time → same column? REJECT
from_date and to_date → same column? REJECT
Duplicate column mapping — multiple row tokens → same column (excluding computed/total tokens)

Type hard failures — numeric token → TEXT column, or text token → INTEGER column

If any check fails, return error to agent with violations list.

Retry Limits in Python (Not System Prompt)
Problem: System prompt says "max 3 attempts" but agent can ignore or miscount.

Solution: ToolContext tracks per-tool call counts. Enforced in Python.

@dataclass
class ToolContext:
    session: ChatSession
    request: Request
    template_id: str | None
    connection_id: str | None
    event_queue: asyncio.Queue
    call_counts: dict[str, int] = field(default_factory=dict)  # tool_name → count
    
    CALL_LIMITS: ClassVar[dict[str, int]] = {
        "validate_pipeline": 3,   # max 3 validation attempts per session turn
        "auto_fix_issues": 3,     # max 3 auto-fix attempts
        "refine_mapping": 5,      # max 5 correction rounds
        "call_qwen_vision": 5,    # max 5 vision calls
    }
    
    def check_call_limit(self, tool_name: str) -> None:
        limit = self.CALL_LIMITS.get(tool_name)
        if limit is None:
            return
        count = self.call_counts.get(tool_name, 0)
        if count >= limit:
            raise CallLimitExceeded(tool_name, count, limit)
        self.call_counts[tool_name] = count + 1
Every tool calls ctx.check_call_limit(tool_name) at entry. CallLimitExceeded is caught in the agent loop and fed back to Qwen as: {"error": "call_limit_exceeded", "tool": "validate_pipeline", "attempts": 3, "limit": 3}. Agent must then tell the user.

State Snapshot Integrity (Cross-Tool Consistency)
Problem: auto_map_tokens → refine_mapping → build_contract — each may use slightly different internal representations (cached vs recomputed). Contract may not reflect latest mapping.

Solution: Every tool reads from and writes to the same persisted state files. No in-memory caching across tool calls.

Rule: Tools operate on disk artifacts, not in-memory state.

Template dir contains the single source of truth:
  {template_dir}/
    chat_session.json          ← pipeline state (session.py)
    chat_history.json          ← conversation history
    report_final.html          ← template HTML
    mapping_pdf_labels.json    ← current mapping
    contract.json              ← current contract
    schema_ext.json            ← DB schema snapshot
Enforcement:

auto_map_tokens writes mapping_pdf_labels.json atomically (write_json_atomic)
refine_mapping reads mapping_pdf_labels.json, modifies, writes atomically
build_contract reads mapping_pdf_labels.json fresh (never cached), builds contract, writes contract.json atomically
validate_pipeline reads contract.json and mapping_pdf_labels.json fresh
No tool holds a reference to a previous tool's return value — each reads from disk
This eliminates cross-tool state drift. If refine_mapping updated the mapping, build_contract sees the update because it reads the file.

Collapsed Tool Chain (Reduced Decision Surface)
Problem: auto_map_tokens, refine_mapping, build_contract form a tight dependency chain. Exposing all individually lets the agent call them in isolation, breaking implicit coupling.

Solution: Add a composite tool that runs the full mapping-to-contract pipeline atomically.

Tool 17: resolve_mapping_pipeline
async def tool_resolve_mapping_pipeline(ctx: ToolContext, template_id: str, connection_id: str) -> dict:
    """
    Atomic pipeline: map → [refine if unresolved] → semantic validate → build contract.
    Reduces agent decision surface. Agent calls ONE tool instead of 3-4.
    """
    # Step 1: Auto-map
    mapping_result = await tool_auto_map_tokens(ctx, template_id, connection_id)
    
    # Step 2: Check completeness
    unresolved = [t for t, v in mapping_result["mapping"].items() if v == "UNRESOLVED"]
    if unresolved:
        return {
            "status": "needs_user_input",
            "unresolved": unresolved,
            "mapping": mapping_result["mapping"],
            "message": f"{len(unresolved)} tokens need manual mapping"
        }
    
    # Step 3: Semantic validation
    violations = _validate_mapping_semantics(mapping_result["mapping"], schema, tokens)
    if violations:
        return {
            "status": "semantic_violations",
            "violations": violations,
            "mapping": mapping_result["mapping"],
            "message": "Mapping has semantic issues"
        }
    
    # Step 4: Build contract
    contract_result = await tool_build_contract(ctx, template_id, connection_id)
    return {"status": "contract_ready", **contract_result}
The individual tools (auto_map_tokens, refine_mapping, build_contract) remain available for fine-grained control, but the system prompt guides the agent to prefer resolve_mapping_pipeline for the normal flow. Individual tools are for when the user specifically asks to change one mapping.

Agent Loop Implementation
Core Loop (hermes_agent.py)
class HermesAgent:
    MAX_TOOL_ROUNDS = 10  # hard safety limit — prevents infinite loops
    
    async def run(self, payload, upload_file=None) -> AsyncIterator[dict]:
        """
        1. Build system prompt with pipeline rules + context
        2. Send to Qwen 3.5 with tool schemas
        3. If response has tool_calls → execute tools → feed results back → goto 2
        4. If response is text (no tool_calls) → yield chat_complete → done
        
        Retry limits enforced in Python via ctx.check_call_limit(), NOT prompt.
        State read from disk on every tool call, NOT cached in memory.
        """
        messages = self._build_messages(payload)
        ctx = ToolContext(session, request, template_id, connection_id, event_queue)
        
        for round in range(MAX_TOOL_ROUNDS):
            response = await call_qwen_with_tools(messages, TOOL_SCHEMAS)
            
            if no_tool_calls(response):
                yield chat_complete(response.content)
                return
            
            for tool_call in response.tool_calls:
                yield stage_event(tool_call.name, "started")
                try:
                    ctx.check_call_limit(tool_call.name)  # Python-enforced retry limit
                    result = await execute_tool(ctx, tool_call)
                except PreconditionError as e:
                    result = {"error": "precondition_failed", "message": str(e)}
                except CallLimitExceeded as e:
                    result = {"error": "call_limit_exceeded", "tool": e.tool, 
                              "attempts": e.count, "limit": e.limit,
                              "message": f"Tool '{e.tool}' called {e.count} times (limit: {e.limit}). Ask user for help."}
                yield stage_event(tool_call.name, "complete")
                messages.append(tool_result(tool_call.id, result))
        
        yield chat_complete("Max rounds reached. Here's where we are...")
System Prompt (hermes_system_prompt.py)
The system prompt tells Qwen:

Identity: "You are NeuraReport, a report-building assistant"
Pipeline states: all 14 states, valid transitions (informational — enforcement is in Python)
Guidance per state (NOT enforcement — tools enforce):
EMPTY: "User must upload a PDF. Call verify_template."
HTML_READY: "Ask user to connect DB or edit. Call auto_map_tokens or resolve_mapping_pipeline when DB connected."
MAPPED: "Show mapping to user. If issues, call refine_mapping. If good, call resolve_mapping_pipeline or build_contract."
APPROVED: "Call validate_pipeline. If errors returned, call auto_fix_issues then validate again. If call_limit_exceeded returned, tell the user."
VALIDATED: "Ask user for date range, then call generate_report."
Conversation style: concise, actionable, show progress
Tool error handling: "When a tool returns an error, explain it to the user and suggest next steps."
What the prompt does NOT enforce (Python enforces these instead):

Retry limits → ctx.check_call_limit() in Python
Mapping completeness → _validate_mapping_semantics() in build_contract
Validation before generation → generate_report checks session.completed_stages
State transitions → session.transition() raises ValueError
Dynamic context injected per turn:

Current state from session.to_dict()
Template summary (tokens, HTML snippet)
DB schema summary
Mapping status (resolved/unresolved counts)
Last validation result
NDJSON Streaming Bridge
Tools push progress events to an asyncio.Queue:

# Inside tool_verify_template:
async for chunk in _iter_streaming(verify_response):
    event = _try_parse_ndjson(chunk)
    if event and event.get("event") == "stage":
        await ctx.event_queue.put(event)  # → streamed to frontend

# Inside agent loop:
while not event_queue.empty():
    yield event_queue.get_nowait()  # drain progress events between tool calls
Frontend sees the same NDJSON stream: chat_start → stage events → chat_complete.

Conversation History (chat_history.py)
class ChatHistory:
    MAX_MESSAGES = 50
    MAX_TOOL_RESULT_CHARS = 2000  # trim old tool results to save context
    
    def append(message: dict) → None
    def get_messages(max_tokens: int = 30000) → list[dict]

Persisted alongside chat_session.json. Loaded on session resume. Sliding window: keeps last N messages, always preserves system prompt. Tool results older than 2 turns get trimmed to summaries.

Risks and Mitigations
Risk	Mitigation
Wrong tool called	Preconditions catch invalid calls in Python; PreconditionError fed back to agent
Tool call explosion (latency)	MAX_TOOL_ROUNDS=10 hard limit; ctx.call_counts per-tool limits
Retry loop ignored/miscounted	ctx.check_call_limit() in Python, NOT prompt. CallLimitExceeded stops tool.
Context window pressure	chat_history.py sliding window; tool results trimmed after 2 turns
Mapping looks correct but is wrong	mapping_sanity.py — type checks, value distribution sampling, cross-field consistency. Hard gate before contract.
Cross-tool state drift	All tools read/write disk artifacts atomically. No in-memory caching across tool calls.
Agent bypasses validation	generate_report checks "validate" in session.completed_stages — Python enforces
GLM-OCR not on Ollama	Create from GGUF; or keep DeepSeek-OCR until available
Plausible wrongness passes all checks	This is the remaining risk. Semantic validation catches type/distribution mismatches but can't catch all semantic errors. Mitigation: dry-run visual comparison (already in validator) + user review step before generation.
What This Plan Does NOT Solve
All 3 candidates consistently wrong: If Qwen maps row_set_wt_kg → bin1_act in ALL 3 candidate attempts (same wrong answer every time), multi-candidate scoring can't help because there's no correct candidate to compare against.

Mitigations (layered, not single-point):

OCR context injects column header text ("Set. wt(Kg)") → name similarity scoring matches "set_wt" to "sp/setpoint" alias
Dry-run visual check compares generated PDF against reference PDF
User reviews sample data in mapping panel before approving
This is a design limit: the system reduces wrong mappings from common to rare, but can't guarantee zero.
